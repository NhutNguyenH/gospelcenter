## Core knowledge (load-bearing)

### jsDelivr facts
- Serves PUBLIC GitHub repos only. Private repos do not work; this is the
  primary gate.
- URL: `https://cdn.jsdelivr.net/gh/{owner}/{repo}@{ref}/{path}` where `{ref}`
  is a branch, tag, or commit SHA.
- Cache TTL: branch refs (`@main`) cached **12 hours** by jsDelivr; tags and
  SHAs are immutable (cached forever, version-pin for free).
- Purge endpoint: `https://purge.jsdelivr.net/gh/{owner}/{repo}/{path}` — open
  in any browser, takes ~10 seconds.
- File size limit: ~50MB. Effectively unlimited for this project.
- Service uptime: high (Cloudflare/Fastly backing). Acceptable as a sole
  dependency for a non-critical widget.

- Verify visibility: `gh repo view NhutNguyenH/gospelcenter --json visibility`
- Verify tracked files: `gh api repos/NhutNguyenH/gospelcenter/contents` or

### Target architecture

- `widget.js` — widget logic (currently inline in `widget-template.html`).
  Hosted on GitHub, served via jsDelivr.
- `translations.js` — pure data (`window.WIDGET_TRANSLATIONS = {...}`). Hosted
  same way. Regenerated by `translate-gemini.js`.

- [ ] Pre-publish secret scan of git **history**
      (`git log --all -p | grep -E "AIzaSy[A-Za-z0-9_-]{30,}"`)?
      Reminder: going public exposes history, not just HEAD.
- [ ] `.env` still gitignored after migration?
- [ ] Plan revokes the previously-leaked key
      (`AIzaSyCuNDMDzIWrM-...`) at https://aistudio.google.com/apikey?
- [ ] No new dependency on a private endpoint or auth token after migration?

### Cache & freshness
- [ ] Purge step in the runbook (not optional — without it, updates wait up to 12h)?
- [ ] Ref strategy declared: `@main` vs pinned tag (`@v1.0`)?
      `@main` = always-fresh + cache, `@v1.0` = rollback-safe but manual bumps.
- [ ] Plan acknowledges that browser HTTP cache is separate from jsDelivr
      cache? After a purge, returning visitors may still see old data until
      they hard-reload.
- [ ] If pinning versions: how does the user create a new tag? Recommend
      `gh release create v1.X.X` or just `git tag v1.X.X && git push --tags`.

### Seamless updates
- [ ] New workflow is **shorter** than the old one? Target:
      `edit local → push → purge URL → done` (no platform login).
- [ ] Plan considers the user is non-technical — `git push` instructions need
      to be explicit (GitHub Desktop OR a `push.bat` script that wraps the
      commands).
- [ ] What happens if jsDelivr is down? (Plan should mention; rare but worth
      one sentence.)
- [ ] What happens if user pushes a broken `translations.js`? (Visitors get
      JS errors. Pinned versions mitigate; `@main` doesn't.)

### Build pipeline alignment
- [ ] `translate-gemini.js` refactor included? Old: writes `widget.html` with
      inline data. New: writes `translations.js` (`window.WIDGET_TRANSLATIONS = {...}`).
- [ ] `widget-template.html` refactor included? Old: contains all JS inline.
      New: HTML shell only; JS extracted to `widget.js`.
- [ ] Is the refactor a separate plan, or part of this one? Either is fine;
      just be explicit.

### Operational
- [ ] Repo description updated to reflect public nature ("Static widget — safe
      to fork")?
- [ ] LICENSE file added if going public? Even MIT is enough; no LICENSE
      means "all rights reserved" which discourages forks/contributions.
- [ ] README explaining what the widget is and how to use it?
